# 🔄 拖延风险评估工具 - 模型升级指南

## 🎯 何时需要升级模型？

### 1. 数据驱动的升级触发点

#### 📊 当收集到足够的新数据时
- **建议阈值**：累计500-1000条新的评估记录
- **为什么**：新数据包含更多真实用户行为模式，能提高模型准确性
- **如何判断**：定期查看 `data/risk_records.db` 中的记录数量

#### 📉 当模型预测准确率下降时
- **表现**：用户反馈预测结果不准确，或实际拖延情况与预测不符
- **如何监测**：
  - 收集用户对预测结果的反馈（已在工具中实现）
  - 统计预测结果与实际拖延情况的匹配度
  - 定期评估模型性能指标

#### 🔄 当数据分布发生变化时
- **表现**：新用户群体与训练数据中的用户特征差异较大
- **例如**：
  - 原本主要是学生使用，现在职场人士使用增多
  - 任务类型发生变化（如从学习任务扩展到工作任务）
  - 时间周期变化（如考试季vs假期）

### 2. 业务驱动的升级触发点

#### 🎯 当评估指标需要调整时
- **例如**：
  - 从单纯预测拖延概率，扩展到预测具体拖延时间
  - 增加新的评估维度（如环境因素、时间压力）
  - 调整评估权重（如更看重自我控制能力）

#### 💡 当获得新的特征工程思路时
- **例如**：
  - 发现任务类型与拖延风险的强相关性
  - 引入时间管理技巧作为新的特征
  - 考虑用户的历史评估记录

#### 🌟 当开源社区提供改进建议时
- **表现**：社区贡献了新的算法、特征或优化方案
- **如何利用**：
  - 定期查看开源仓库的Issue和Pull Request
  - 测试社区贡献的改进方案
  - 合并有效的改进到主分支

## 📋 模型升级流程

### 1. 数据准备阶段

#### 📥 收集和整理数据
- **现有数据**：`data/risk_records.db` 中的历史记录
- **新数据来源**：
  - 用户的评估记录
  - 用户反馈
  - 实际拖延情况的跟踪数据

#### 🧹 数据清洗和预处理
- 移除异常值（如评分超出0-100范围的数据）
- 处理缺失值（如某些字段为空的数据）
- 标准化数据格式
- 划分训练集和测试集

### 2. 模型训练阶段

#### 🔧 选择合适的算法
- **当前算法**：逻辑回归（简单、高效、易解释）
- **可选算法**：
  - 随机森林（处理复杂关系）
  - 支持向量机（处理高维数据）
  - 梯度提升树（更高的预测准确率）

#### 📝 调整模型参数
- 尝试不同的特征权重
- 调整正则化参数
- 优化训练超参数

#### 🧪 训练新模型
- 使用 `train_model.py` 脚本训练新模型
- 保存为 `new_model.pkl` 或覆盖原有模型

### 3. 模型评估阶段

#### 📊 评估模型性能
- **准确率**：预测结果与实际情况的匹配度
- **召回率**：正确识别出高风险用户的比例
- **F1分数**：准确率和召回率的综合指标
- **AUC-ROC**：模型区分正负样本的能力

#### 🆚 比较新旧模型
| 指标 | 旧模型 | 新模型 | 是否改进 |
|------|--------|--------|----------|
| 准确率 | 0.85 | 0.89 | ✅ |
| 召回率 | 0.78 | 0.83 | ✅ |
| F1分数 | 0.81 | 0.86 | ✅ |

### 4. 模型部署阶段

#### 🚀 部署新模型
- **本地部署**：直接替换 `model.pkl` 文件
- **云服务器部署**：
  ```bash
  # 上传新模型
  scp new_model.pkl root@服务器IP:/var/www/procrastination-tool/model.pkl
  
  # 重启服务
  sudo systemctl restart procrastination-tool
  ```

#### 📈 监控新模型表现
- 收集用户反馈
- 跟踪预测准确率
- 分析预测误差
- 准备回滚方案（如新模型表现不佳，立即切换回旧模型）

### 3. 模型评估脚本

**创建简单的评估脚本**：
```python
#!/usr/bin/env python3
"""模型评估脚本"""

import pickle
import numpy as np
from sklearn.metrics import accuracy_score, recall_score, f1_score, roc_auc_score

# 加载模型
with open('model.pkl', 'rb') as f:
    model = pickle.load(f)

# 加载测试数据（需要准备）
test_data = np.load('test_data.npy')
test_labels = np.load('test_labels.npy')

# 预测
predictions = model.predict(test_data)

# 评估
accuracy = accuracy_score(test_labels, predictions)
recall = recall_score(test_labels, predictions)
f1 = f1_score(test_labels, predictions)
auc = roc_auc_score(test_labels, predictions)

print(f"准确率: {accuracy:.4f}")
print(f"召回率: {recall:.4f}")
print(f"F1分数: {f1:.4f}")
print(f"AUC-ROC: {auc:.4f}")
```

## 🧠 简单决策框架：是否升级模型？

### 快速判断法

**回答以下3个问题**：
1. ⏰ 是否收集了足够的新数据？（建议500+条）
2. 📉 是否有用户反馈预测不准确？
3. 🔄 是否发现评估数据的分布发生了变化？

**如果至少有1个问题回答「是」**，则考虑升级模型。

### 详细判断法

| 指标 | 升级阈值 | 建议行动 |
|------|----------|----------|
| 新数据量 | ≥500条 | 升级模型 |
| 用户反馈 | ≥10%的用户反馈不准确 | 升级模型 |
| 预测准确率 | 下降≥5% | 升级模型 |
| 数据分布变化 | 特征均值变化≥10% | 升级模型 |
| 开源社区贡献 | 有更好的算法或特征 | 升级模型 |

## 🎯 模型升级的最佳实践

### 1. 保持模型简单
- **优先选择**：简单、易解释的模型（如逻辑回归）
- **避免**：过于复杂的模型（如深度神经网络）
- **原因**：简单模型更稳定，不易过拟合，易解释

### 2. 重视用户反馈
- **设置反馈渠道**：在工具中收集用户对预测结果的反馈
- **分析反馈数据**：找出预测不准确的模式
- **结合用户反馈改进特征**：根据反馈调整特征权重或添加新特征

### 3. 逐步迭代升级
- **小步快跑**：每次只做一个小的改进
- **A/B测试**：同时运行新旧模型，比较表现
- **监控效果**：升级后密切监控模型表现，准备回滚方案

### 4. 利用开源社区力量
- **鼓励社区贡献**：邀请社区成员提交改进建议
- **测试社区贡献**：验证社区提供的改进方案
- **合并有效改进**：将优秀的社区贡献合并到主分支

### 5. 记录升级历史
- **保存旧模型**：将旧模型备份为 `model_YYYYMMDD.pkl`
- **记录改进点**：详细记录每次升级的原因、改进内容和效果
- **撰写升级日志**：在项目README或CHANGELOG中记录升级历史

## 🎉 总结

模型升级是一个**持续迭代**的过程，没有「一劳永逸」的模型。通过**定期收集数据**、**倾听用户反馈**、**监控模型表现**，您可以持续改进模型，提供更准确、更有用的评估结果。

3. **分布变化了**：新用户群体不同了
4. **算法改进了**：有更好的算法或特征
5. **社区贡献了**：开源社区提供了改进方案
